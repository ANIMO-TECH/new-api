本文档用于说明最近 3 个 commit 引入/调整的机制，供测试人员执行功能测试与回归测试。

## 对应提交

- `18bf4998`：增加模型级告警配置，包括“渠道不足告警阈值”和“长期禁用时告警”

## 功能概览

本次改动涉及 3 组机制：

1. 自动禁用后的指数退避
2. 到期后自动复活通道
3. 模型级告警

### 1. 自动禁用后的指数退避

当通道因为真实请求失败、批量测试失败、余额不足等原因被“自动禁用”时：

- 系统会记录本次自动禁用次数
- 如果开启了“到期后自动复活通道”，则会计算本次禁用持续时间
- 禁用持续时间按指数退避公式计算：

```text
本次禁用时长 = 首次禁用时长 * 退避倍率 ^ (自动禁用次数 - 1)
```

例如：

- 首次禁用时长 = `60 秒`
- 退避倍率 = `2`

则第 1/2/3 次自动禁用后的禁用时长分别为：

- 第 1 次：60 秒
- 第 2 次：120 秒
- 第 3 次：240 秒

### 2. 到期后自动复活通道

该机制独立于原有“成功时自动启用通道”。

- 原有“成功时自动启用通道”仍主要用于自动测试链路
- 新增“到期后自动复活通道”用于指数退避后的惰性自动复活

工作方式：

- 通道处于 `auto_disabled` 状态时，不会由后台定时扫描恢复
- 当有新请求再次尝试选择该模型对应的通道时，系统会检查该通道是否已到复活时间
- 如果已到期，且未超过最大自动复活次数，则自动恢复为启用状态
- 如果已经达到最大自动复活次数，则不再自动复活，通道保持自动禁用

### 3. 模型级告警

为模型元数据增加了 2 个配置：

- `渠道不足告警阈值`
- `长期禁用时告警`

#### 渠道不足告警阈值

含义：

- 当某个模型在某个分组下的“剩余启用渠道数”小于该值时，发送通知给 root 用户

规则说明：

- `0` 表示关闭该告警
- `1` 表示仅在剩余启用渠道数 `< 1` 时告警，即没有可用渠道时告警
- `2` 表示剩余启用渠道数为 `0` 或 `1` 时告警

注意：

- 这是按“模型 + 分组”维度判断的，不是全局模型维度
- 只对“启用中的模型配置”生效；如果模型元数据 `status != 1`，不会参与告警匹配

#### 长期禁用时告警

含义：

- 当关联该模型的某个通道达到最大自动复活次数、后续不再自动复活时，发送通知给 root 用户

规则说明：

- 该配置也是模型级开关
- 通知实际是由“通道达到最大自动复活次数”这一事件触发
- 同一个通道在同一轮长期禁用周期内只会发送一次该告警，不会重复刷通知
- 当通道再次进入新的自动禁用周期，或被手动启用后重新开始计数时，才可能再次触发

## 管理后台位置

### 监控设置

路径：

- 管理后台
- 系统设置
- 运营设置
- 监控设置

涉及开关/参数：

- `失败时自动禁用通道`
- `成功时自动启用通道`
- `到期后自动复活通道`
- `首次禁用时长`
- `退避倍率`
- `最大复活次数`

说明：

- `到期后自动复活通道` 必须开启，退避参数才有实际作用
- 当 `最大复活次数 = 0` 时，不自动复活，行为等同于“直接自动禁用”

### 模型配置

路径：

- 管理后台
- 模型管理
- 新建/编辑模型

涉及字段：

- `渠道不足告警阈值`
- `长期禁用时告警`

## 推荐测试前置条件

建议测试前准备：

- 至少 2 个相同模型、相同分组的可用通道
- 至少 1 个可稳定复现失败的通道
- root 用户已经配置可接收通知的方式
  - 邮件 / Webhook / Bark / Gotify 任意一种
- 确认自动禁用总开关已开启

建议准备 2 类模型配置：

1. 精确匹配模型
   - 例如：`gpt-4o-mini`

2. 规则匹配模型
   - 例如：前缀、后缀、包含匹配

用于验证模型告警匹配逻辑是否正确。

## 测试项

### A. 自动禁用基础回归

目标：

- 确认原有自动禁用仍能正常工作

步骤：

1. 开启 `失败时自动禁用通道`
2. 配置一个可稳定失败的通道
3. 发起真实请求，触发自动禁用

预期：

- 通道状态变为自动禁用
- 禁用原因被记录
- 不影响原有手动禁用、手动启用逻辑

### B. 指数退避参数生效

目标：

- 确认自动禁用次数会影响禁用持续时间

步骤：

1. 开启：
   - `失败时自动禁用通道`
   - `到期后自动复活通道`
2. 设置：
   - `首次禁用时长 = 60`
   - `退避倍率 = 2`
   - `最大复活次数 = 3`
3. 让同一个通道连续多次进入“自动禁用 -> 到期 -> 自动复活 -> 再次失败”

预期：

- 第 1 次禁用后，约 60 秒可复活
- 第 2 次禁用后，约 120 秒可复活
- 第 3 次禁用后，约 240 秒可复活

注意：

- 当前是“惰性自动复活”，需要在到期后再次发请求/触发选通道，系统才会尝试复活

### C. 惰性自动复活

目标：

- 确认系统不会靠后台任务自动扫库复活，而是在选通道时复活

步骤：

1. 让目标通道进入自动禁用状态
2. 等待超过禁用时长
3. 不发任何请求，观察通道状态
4. 然后发起一次使用同组同模型的请求

预期：

- 第 3 步：通道仍可能保持 `auto_disabled`
- 第 4 步：请求触发选通道时，系统尝试复活该通道
- 若未超过最大复活次数，则通道恢复为启用

### D. 最大复活次数 = 0

目标：

- 确认“0 表示不自动复活”

步骤：

1. 开启 `失败时自动禁用通道`
2. 开启 `到期后自动复活通道`
3. 设置 `最大复活次数 = 0`
4. 触发通道自动禁用
5. 等待超过原本应有的禁用时长
6. 再次发请求

预期：

- 通道不会被自动复活
- 行为等同于原先“直接自动禁用”

### E. 达到最大复活次数后停止自动复活

目标：

- 确认通道在复活次数耗尽后保持自动禁用

步骤：

1. 设置：
   - `首次禁用时长 = 30`
   - `退避倍率 = 2`
   - `最大复活次数 = 1`
2. 让通道经历：
   - 第 1 次自动禁用
   - 到期后自动复活
   - 再次失败自动禁用
3. 再次等待超过禁用时长，并发请求

预期：

- 通道不会再次自动复活
- 通道保持自动禁用

### F. 手动启用会重置自动禁用状态

目标：

- 确认手动启用会清空自动禁用相关计数和标记

步骤：

1. 让通道经历至少 2 次自动禁用，确保退避时长已增大
2. 在管理后台手动启用该通道
3. 再次触发自动禁用

预期：

- 通道重新按“首次禁用时长”开始计算
- 不应延续旧的自动禁用次数和长期禁用通知状态

### G. 渠道不足告警阈值 = 0

目标：

- 确认阈值为 0 时，不发送该类告警

步骤：

1. 模型 `A` 配置：
   - `渠道不足告警阈值 = 0`
2. 该模型所在分组准备 2 个启用通道
3. 禁用其中 1 个甚至全部

预期：

- 不发送“模型可用渠道不足”通知

### H. 渠道不足告警阈值生效

目标：

- 确认按“分组 + 模型”维度判断剩余启用渠道数

步骤：

1. 模型 `A` 配置：
   - `渠道不足告警阈值 = 2`
2. 在分组 `default` 下准备 2 个启用通道，都支持模型 `A`
3. 自动禁用其中 1 个

预期：

- 因剩余启用渠道数为 `1`
- 且 `1 < 2`
- 发送“模型可用渠道不足”通知

通知内容应包含：

- 通道名称 / ID
- 模型名称
- 分组名称
- 当前剩余启用渠道数
- 告警阈值

### I. 分组维度独立判断

目标：

- 确认不是按全局模型数判断，而是按“模型在某个分组下的剩余启用渠道数”判断

步骤：

1. 模型 `A` 同时存在于分组 `group1`、`group2`
2. `group1` 下仅 1 个启用通道
3. `group2` 下有 3 个启用通道
4. `渠道不足告警阈值 = 2`
5. 禁用 `group1` 下唯一通道

预期：

- 应对 `group1` 触发告警
- 不应因为 `group2` 仍有足够通道而吞掉告警

### J. 规则匹配模型告警

目标：

- 确认前缀/后缀/包含匹配的模型元数据也能触发告警

步骤：

1. 新建一个规则匹配模型配置
   - 例如前缀匹配：`gpt-4o`
2. 设置：
   - `渠道不足告警阈值 = 2`
3. 使用运行时模型名：
   - `gpt-4o-mini`
4. 减少该模型在某个分组下的可用通道数

预期：

- 规则匹配生效
- 发送对应告警

### K. 停用的模型配置不参与告警

目标：

- 确认模型元数据 `status != 1` 时，不参与告警匹配

步骤：

1. 某模型配置设置：
   - `渠道不足告警阈值 = 2`
2. 将该模型配置状态改为停用
3. 再次触发对应通道自动禁用

预期：

- 不发送该模型的告警通知

### L. 长期禁用时告警

目标：

- 确认达到最大复活次数后发送一次长期禁用通知

步骤：

1. 模型配置开启：
   - `长期禁用时告警 = 开`
2. 通道配置：
   - `最大复活次数 = 1`
3. 让通道经历：
   - 自动禁用
   - 自动复活
   - 再次自动禁用
4. 等待到期后再次发请求，触发“尝试复活但次数已耗尽”

预期：

- 发送“长期禁用”通知
- 通知内容包含：
  - 通道名称 / ID
  - 最大自动复活次数
  - 受影响模型与分组

### M. 长期禁用通知只发一次

目标：

- 确认同一轮长期禁用周期不会重复刷通知

步骤：

1. 先完成用例 L
2. 在不手动启用通道的情况下，多次继续发请求

预期：

- 通道仍保持自动禁用
- 不重复发送同一通道的长期禁用通知

### N. 手动启用后可重新触发长期禁用通知

目标：

- 确认长期禁用“一次性通知标记”会被重置

步骤：

1. 先完成用例 L
2. 手动启用该通道
3. 再次让该通道走完整个“自动禁用 -> 自动复活 -> 再次禁用 -> 次数耗尽”的流程

预期：

- 可以再次发送长期禁用通知

## 回归关注点

## 已知实现特征

以下是当前实现特征，测试时可以按此理解，不属于缺陷：

- 自动复活是“惰性复活”，不是定时扫描复活
- 模型告警是按“模型 + 分组”维度判断
- 规则匹配模型也会参与告警
- 前端新增文案尚未执行 i18n 同步脚本，非中文环境可能先回退显示中文
